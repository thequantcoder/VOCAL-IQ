# WhatsApp Calling — SIP mode (Asterisk) runbook (WAC-10)

> **Optional / opt-in per number.** Use this only for a tenant that already runs a SIP PBX
> (Asterisk/Kamailio) and wants their WhatsApp number's calls bridged through it instead of the default
> Graph-API + WebRTC path (WAC-00..09). SIP mode is **mutually exclusive** with Graph-API calling for
> that number — enabling it strips the `calls` webhook (no SDP in webhooks). See
> `docs/WHATSAPP-CALLING-AI-ENGINE-PLAN.md` §A.2/§A.6/§A.8.

## Prerequisites

- A **TLS SIP server** reachable from the internet on **5061**, with a **valid public certificate**
  (Meta acts as the TLS *client* and will **not** present a client cert — **do NOT require mTLS**).
- Digest auth enabled. Realm is **`wa.meta.vc`**; username is the **business phone number**; the
  password is **generated by Meta** (fetch it — see below).
- The tenant's app in **Live** mode, and an explicit decision to run this number in SIP mode.

## Enable SIP mode

1. Configure via the API (config-writer): `PUT /whatsapp-calling/sip`
   ```json
   {
     "enabled": true,
     "servers": [{ "hostname": "pbx.acme.example", "port": 5061 }],
     "webhookDelivery": false,
     "srtpProtocol": "SDES"
   }
   ```
   This syncs `calling.sip` to Meta and persists it. (`srtpProtocol`: `DTLS` or `SDES` — Asterisk chan_pjsip
   with `media_encryption=sdes` is simplest.)
2. Fetch the Meta-generated digest password: `GET /whatsapp-calling/sip/credentials`
   (proxies `GET /settings?include_sip_credentials=true`). Store it in the PBX auth — **never log it**.
3. **Reset procedure** (rotate creds / change servers): disable SIP → clear `servers` → re-enable →
   re-fetch credentials. Meta regenerates the password on re-enable.

## Signaling

- **Outbound**: `INVITE sip:+<number>@wa.meta.vc;transport=tls` → `407 Proxy Authentication Required`
  → re-INVITE with digest auth (username = business number, realm = `wa.meta.vc`, Meta password) → SDP
  offer (OPUS; ICE/DTLS-SRTP or SDES). The **business side sends the first SRTP packet**.
- **Inbound**: Meta sends the `INVITE` to your SIP server; answer with `200 OK` + SDP answer.
- Context + correlation arrive as **`x-wa-meta-*`** headers (there is **no** `calls` webhook with SDP):
  `x-wa-meta-wacid` (= WACID → correlate to Calls), `x-wa-meta-user-id`,
  `x-wa-meta-cta-payload` / `x-wa-meta-deeplink-payload` (user-initiated context), and on **BYE**
  `x-wa-meta-call-duration` (→ meter via WAC-06). VocalIQ parses these with `parseWaMetaHeaders`.

## ⚠️ The Record-Route / ACK gotcha (calls drop at ~32 s)

Asterisk's default `rewrite_contact=yes` rewrites the Contact to the source IP:port, which breaks the
in-dialog **ACK** routing to Meta — the call is torn down at roughly **32 seconds**. **Fix: set
`rewrite_contact=no`** on the WhatsApp endpoint (Meta is a well-formed proxy that sets Record-Route
correctly), and honour Record-Route so the ACK/BYE follow the signalled path.

## Reference config (chan_pjsip, SDES)

```ini
; pjsip.conf
[transport-tls]
type=transport
protocol=tls
bind=0.0.0.0:5061
cert_file=/etc/asterisk/keys/fullchain.pem
priv_key_file=/etc/asterisk/keys/privkey.pem
method=tlsv1_2
; Meta is the TLS client and presents NO cert — do not require one:
require_client_cert=no
verify_client=no

[wa-endpoint](!)
type=endpoint
context=whatsapp-in
disallow=all
allow=opus
media_encryption=sdes
rtp_symmetric=yes
force_rport=yes
rewrite_contact=no        ; <-- the ACK fix (else calls drop ~32s)

[wa](wa-endpoint)
type=endpoint
aors=wa
outbound_auth=wa-auth

[wa-auth]
type=auth
auth_type=userpass
username=<BUSINESS_NUMBER>     ; e.g. 16315553601
password=<META_GENERATED>      ; from GET /whatsapp-calling/sip/credentials
realm=wa.meta.vc

[wa]
type=aor
contact=sip:wa.meta.vc:5061;transport=tls
```

```ini
; extensions.conf
[whatsapp-in]
exten => _+X.,1,NoOp(WhatsApp SIP call ${SIP_HEADER(X-WA-Meta-WACID)})
 same => n,Answer()
 same => n,Stasis(vocaliq-ai)     ; bridge to the AI loop (or Dial your agents)
 same => n,Hangup()
```

```ini
; rtp.conf
[general]
rtpstart=10000
rtpend=20000
strictrtp=yes
```

## Validate

- TLS reachability: `openssl s_client -connect pbx.acme.example:5061 -tls1_2` — must complete a handshake
  presenting a **valid public cert** and must **not** demand a client cert.
- Place a test WhatsApp call; confirm the `x-wa-meta-wacid` header appears and the call survives past
  32 s (proves the ACK fix). On BYE, confirm a `UsageRecord` was written for the duration.

## Rollback

Disable SIP mode (`PUT /whatsapp-calling/sip {"enabled": false}`) → the number returns to Graph-API +
WebRTC calling (WAC-00..09). No data migration needed.
